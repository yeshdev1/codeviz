A read request3 steps · Clients → API → Data & workers▶
When an app shows you a list — your orders, your messages — it first has to fetch that data from the server. This walks one read request from the app down to where the data lives, and shows a trick servers use to stay fast: checking a cache before the database. No prior knowledge assumed; every term is defined as it appears.
Step 1 of 3
Client→APIGET /things

Your app does not read the database directly. It asks the API — a single server program that owns the rules and is the only thing allowed to touch the database — to fetch the data for it, using an HTTP request.

Why route everything through one API? Because the database password and the business rules then live in exactly one place, instead of being copied into every app (web, mobile, desktop). One door, one set of locks.

New to this?An API is a server that other programs talk to over HTTP. GET means “fetch something” — this request only reads data, it never changes anything.
API — a server that other programs call over HTTP to fetch or change dataHTTP request — a message asking a server to do or return somethingclient — the app making the request — a browser, phone, or script
Step 2 of 3
API→Cachecache lookup

Before doing the slow thing — reading the database — the API checks a cache first. A cache is a small, very fast store that keeps recently-used answers in memory.

If the answer is already there (a cache hit) the API can reply in well under a millisecond and never bothers the database at all — that is why this is the fastest hop on the map. This time imagine it is a miss: the answer is not cached yet, so we go one layer deeper.

New to this?A cache trades freshness for speed: it keeps copies of recent results in memory so common reads can skip the database. A hit means the copy was there; a miss means it was not.
cache — a fast in-memory store of recent results, used to avoid slow lookupscache hit / miss — whether the cache already had the answer (hit) or not (miss)in-memory — kept in RAM, which is far faster to read than disk
Step 3 of 3
API→DatabaseSELECT …

On a miss, the API reads from the database — the single, permanent source of truth where the real data lives in tables of rows. It pulls out just the rows it needs, then quietly saves the result back into the cache so the next identical request will be a fast hit.

The query is parameterised: the values you searched for are sent to the database separately from the query text. That sounds like a detail, but it is exactly what prevents SQL injection — a classic attack where malicious input is smuggled in as commands.

New to this?The database is the authoritative copy of your data, kept on disk so it survives restarts. A parameterised query keeps user input as data, never as part of the command — which is what makes it safe.
database — the permanent source of truth, storing data in tables of rowsquery — a request to the database to read or change specific rowsparameterised query — a query where inputs are sent separately from the text — safe from injectionSQL injection — an attack that sneaks commands in through input; parameterising stops it
A write + background job4 steps · Clients → API → Data & workers → External▶
Reading data is only half the story — sometimes you change something (place an order, post a comment). Writes are riskier: they must be saved safely, and they often kick off slow follow-up work (emails, charging a card) that you should not have to wait for. This walks a write and shows how the system keeps it both safe and fast.
Step 1 of 4
Client→APIPOST /orders

Just like reads, a change goes through the same single API. The app sends a POST request — the HTTP verb for “create or submit” — carrying the new order's details.

Funnelling writes through one place means the API can check every change in one spot: are you allowed to do this, is the data valid, does it break any rules. You never want that logic scattered across a dozen apps.

New to this?POST is the HTTP verb for creating or submitting something (versus GET for fetching). Routing writes through one API keeps validation and permissions in a single place.
POST — the HTTP verb for creating or submitting datavalidation — checking that incoming data is well-formed and allowed before saving it
Step 2 of 4
API→DatabaseINSERT … (transaction)

The API saves the order into the database inside a transaction — an all-or-nothing wrapper around a group of writes.

Why? Creating an order is usually several writes at once (the order, its line items, a total). If the server crashed halfway, you must never be left with half an order. A transaction guarantees that either every write lands or none do — there is no in-between, even if two requests race.

New to this?A transaction bundles several database changes so they all commit (become permanent) together or all roll back (undo) together. There is no half-finished state.
transaction — a group of writes that all succeed or all fail togethercommit / roll back — make the changes permanent, or undo them entirelydurable — once committed, the data survives crashes and restarts
Step 3 of 4
API→Workerenqueue job

Now the slow part: we may need to email a receipt and charge a card. Those talk to outside services that are slow and can fail, so the API does not do them while you wait.

Instead it drops a note onto a queue — a to-do list — and returns to you immediately. Your order looks done in a split second. This hop is fire-and-forget: nothing comes back on it.

New to this?A queue is a to-do list that background workers pull from. Enqueueing a job lets the API answer you now and let the slow work happen later, out of sight.
queue — a to-do list of jobs for background workers to process laterenqueue — add a job to the queue and move onfire-and-forget — start a job and continue without waiting for a result
Step 4 of 4
Worker→3rd partycall provider (retry)

Moments later a background worker picks the job off the queue and calls the third-party provider (the payment or email service). This happens asynchronously — in the background, long after you already saw your confirmation.

If the provider is slow or down, the worker simply tries again later with backoff (waiting longer between attempts). Because it is off the request path, none of that flakiness can ever slow you down.

New to this?Asynchronous work happens in the background; the user is never blocked on it. Retry with backoff means re-attempting a failed job later, waiting longer each time, so a struggling service is not hammered.
background worker — code that processes queued jobs out of sight of the userasynchronous — happening in the background; the caller does not waitretry with backoff — re-trying a failed job later, with growing delays
