Database

Postgres4 entities · 3 relationships · detail: standard · order_items is the junction

This is the relational database — the permanent, authoritative home for the app's core data. “Relational” means the data is split across several focused tables that relate to each other, rather than one giant blob; that split is what the diagram below shows.

There are four tables here: users (the people), products (the catalogue), orders (a purchase someone made), and order_items (the individual lines within an order). Read the boxes first, then the lines between them.

How this data model works

How to read the boxes. Each box is a table — one kind of record, stored as rows like a spreadsheet. The first column, marked PK (primary key), is the row's unique id. A column marked FK (foreign key) holds the id of a row in another table — that is how two tables get linked. On the diagram an FK is a line ending in a crow's foot (the “many” side) and a bar (the “one” side).

The relationships, in plain English.

  • A user has many orders. Each orders row carries a user_id pointing back to one users row, so one person can place many orders, but every order belongs to exactly one person. (one-to-many)
  • An order has many line items. Each order_items row has an order_id pointing at its parent order. So an order with three products is one orders row plus three order_items rows. (one-to-many)
  • Each line item names one product. order_items.product_id points at a products row. The same product can appear on thousands of line items. (many-to-one)

Why split it up like this? Notice order_items sits between orders and products. It is a join table (or “junction”): orders and products have a many-to-many relationship — an order contains many products, and a product appears in many orders — and a relational database can only express that by putting a table in the middle. It also has somewhere natural to record qty and the price_cents at the moment of purchase, so later edits to the product's price never rewrite history.

The big idea: normalisation. Each fact lives in exactly one place — a product's name is stored once in products, not copied onto every order. The FK lines are how you stitch those facts back together when you read (“for this order, follow order_items to products to get the names”). That avoids duplicated, drifting data, which is the whole point of a relational schema.

How this data model works

Read this diagram like a map of how the records connect. Each box is a table — one kind of thing, stored as rows (think of a spreadsheet). PK marks the column that gives each row a unique identity; FK marks a column that points at another table's row. Every FK is drawn as a line that ends in a crow's foot (the “many” side) and a single bar (the “one” side): so many rows on the foot side belong to one row on the bar side.

The relationships here:

  • Each session:{token} references a users that lives in another store — via session:{token}.user_id (shown as an ext badge, no line drawn here).
  • Each cache:user:{id} references a users that lives in another store — via cache:user:{id}.id (shown as an ext badge, no line drawn here).
